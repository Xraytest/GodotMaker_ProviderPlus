# Project Layout

This page describes the directory structure of a game project generated by GodotMaker.

## Overview

```
my-game/
  project.godot
  CLAUDE.md
  GDD.md
  PLAN.md
  STRUCTURE.md
  ASSETS.md
  SCENES.md
  MEMORY.md
  TOC.md
  components/
  systems/
  entities/                  (optional)
  test/
  tests/e2e/
  addons/
    gecs/
    gdunit4/
    godot_e2e/
  assets/
    sprites/
    audio/
    fonts/
    ui/
  references/
  screenshots/
  tools/
  .claude/
    skills/
    config/
    templates/
    settings.json
    godotmaker.yaml
  .godotmaker/
    hooks/
    config.yaml
    stage_schemas.json
    version
```

## Root Documents

| File | Purpose |
|------|---------|
| `project.godot` | Godot project configuration. Defines the main scene, display settings, and autoloads. |
| `CLAUDE.md` | AI instructions specific to this game project. Controls how Claude Code behaves when working in this directory. |
| `GDD.md` | Game Design Document. Produced during the planning phase. Describes gameplay, mechanics, art direction, and scope. |
| `PLAN.md` | Task breakdown used to track generation progress. |
| `STRUCTURE.md` | ECS architecture document. Lists all Components (with fields), Systems (with read/write declarations), and Scene definitions. |
| `ASSETS.md` | Asset manifest. Maps logical asset names to file paths and records generation parameters. |
| `SCENES.md` | Scene definitions. Describes marker nodes in each scene and their mapping to ECS entity archetypes. |
| `MEMORY.md` | Maintains context across sessions so GodotMaker can resume work consistently. |
| `TOC.md` | Table of contents for all project documents. Updated at the end of generation. |

## .claude/ -- AI Configuration

This directory is managed by GodotMaker. You do not need to edit it.

| Path | Purpose |
|------|---------|
| `skills/` | AI skill definitions used during game generation. |
| `config/` | Shared configuration files used by hooks and skills. |
| `templates/` | Document templates for planning and tracking. |
| `settings.json` | Claude Code settings for this project (permissions, hook registrations, MCP server config). |
| `godotmaker.yaml` | GodotMaker-specific settings: Godot executable path, project preferences. |

## .godotmaker/ -- Pipeline State

This directory tracks the orchestrator's pipeline state and hooks. Core files (`hooks/`, `config.yaml`, `stage_schemas.json`, `version`) are committed to git for version tracking. Runtime state files are gitignored and managed automatically -- you do not need to edit them.

| File | Purpose |
|------|---------|
| `hooks/` | Pipeline scripts. Managed automatically. |
| `config.yaml` | Project-level configuration (e.g., `vqa_model: gemini-3-flash`). |
| `stage_schemas.json` | Stage validation rules. Managed automatically. |
| `version` | GodotMaker version that published this project. Used to detect when republishing is needed. |

## Game Code

### components/

ECS component definitions. Each file defines one or more component classes extending gecs `Component`.

```
components/
  velocity_component.gd
  bounce_component.gd
  sprite_component.gd
  ...
```

Components are pure data containers -- they hold fields but no logic. Systems read and write component data.

### systems/

ECS system implementations. Each file defines a system class extending gecs `System`.

```
systems/
  movement_system.gd
  bounce_system.gd
  render_system.gd
  ...
```

Each system declares which components it reads and writes. GodotMaker verifies there are no conflicting writes between systems.

### entities/ (optional)

Entity archetype definitions, if the game uses predefined entity templates. Not all projects use this directory -- some define archetypes inline in scene markers.

### test/

Unit tests using gdUnit4. Follows TDD -- tests are generated before or alongside implementation.

```
test/
  movement_system_test.gd
  bounce_system_test.gd
  ...
```

Run tests via:
```bash
godot --headless -s addons/gdunit4/bin/gdunit4_run.gd --single --file res://test/movement_system_test.gd
```

### tests/e2e/

End-to-end tests using godot-e2e. These tests run the actual game, capture screenshots, and verify gameplay behavior.

```
tests/e2e/
  bounce_e2e.gd
  ...
```

## addons/

Third-party Godot addons required by every GodotMaker project.

| Addon | Purpose |
|-------|---------|
| `gecs/` | ECS framework. Provides Entity, Component, System, and World classes. |
| `gdunit4/` | Unit testing framework. Provides assertions, test runners, and CLI integration. |
| `godot_e2e/` | End-to-end testing framework. Provides game runner, screenshot capture, and assertion helpers. |

## assets/

Game assets organized by type.

| Directory | Contents |
|-----------|----------|
| `sprites/` | 2D sprite images (PNG). Generated by Gemini or provided by the user. |
| `audio/` | Sound effects and music files. |
| `fonts/` | Custom fonts (TTF/OTF). |
| `ui/` | UI-specific assets (icons, panel backgrounds, button styles). |

## references/

Design reference images. The orchestrator generates `reference.png` during the asset collection phase -- a visual target showing what the game should look like. Visual QA compares screenshots against this reference.

## screenshots/

Gameplay screenshots captured during the verification phase. Used by visual QA to assess whether the game matches the reference image.

## tools/

Project-level utility scripts.

| Script | Purpose |
|--------|---------|
| `check_project.py` | Validates project structure, required files, and addon presence. Run by verifiers during integration. |
| `check_env.py` | Checks that all required tools and API keys are available. |
