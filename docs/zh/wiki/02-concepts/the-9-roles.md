# 9 个角色

每个角色对应一条你在 Claude Code 里输入的斜杠命令。按顺序执行——如果跳过了某个前置步骤，系统会告诉你。

这些命令分为两种序列。`/gm-scaffold` 在项目最开始只运行一次。从 `/gm-gdd` 到 `/gm-finalize` 构成一个"里程碑"——一轮完整的设计、构建、交付流程。`/gm-finalize` 关闭一个里程碑之后，你可以用下一个 `/gm-gdd` 开启新的里程碑。

---

## `/gm-scaffold`

**做什么：** 创建一个空的 Godot 项目，建好正确的目录结构，安装必需的插件（`gecs` 提供 ECS 支持，`gdUnit4` 提供测试支持），并提交第一个 git commit。

**什么时候运行：** 在一切开始之前，只运行一次。项目目录必须已存在但是空的（或者只有一个 `.git` 文件夹）。

**背后发生了什么：**
- 写入 `project.godot`、`addons/`、`src/`、`scenes/`、`assets/`、`e2e/`、`tests/`
- 安装并配置 `gecs` 和 `gdUnit4`
- 创建 `e2e/conftest.py`（测试框架入口）

**你得到什么：** 一个干净、可运行的 Godot 项目，还没有任何游戏逻辑。

**需要知道的：** 这条命令每个项目只运行一次。在同一个项目里再次运行会提前中止并给出说明。

---

## `/gm-gdd`

**做什么：** 问你想做什么游戏，然后写出后续所有步骤都要参考的策划文档。

**什么时候运行：** 在 `/gm-scaffold` 之后（第一个里程碑），或在 `/gm-finalize` 之后（后续每个里程碑）。

**背后发生了什么：**
- 向你提问：游戏类型、机制、美术风格、规模、平台
- 写出 `GDD.md`（设计合同）、`PLAN.md`（任务清单）、`STRUCTURE.md`（目录与架构规划）、`SCENES.md`（逐场景描述）、`ASSETS.md`（所需美术和音频）以及 `TOC.md`（目录索引）

**你得到什么：** 一套完整的文档，构建、评估、评审步骤都会以这套文档为准，检验游戏是否达标。

**需要知道的：** 这一步的质量直接决定最终成果的质量。描述越具体越好——"俯视角僵尸射击，波次刷怪 + 高分榜"比"一个僵尸游戏"有用得多。运行 `/gm-asset` 之前你可以自己编辑这些文档。

---

## `/gm-asset`

**做什么：** 确保构建开始前，`ASSETS.md` 里列出的每个美术和音频文件都真实存在；并为 `SCENES.md` 描述的每个场景生成一张视觉参考图。

**什么时候运行：** 在 `/gm-gdd` 之后、`/gm-build` 之前。

**背后发生了什么：**
- 读取 `ASSETS.md`，找出游戏需要的所有资源
- 对于你已经提供的资源：派一个 Analyst（分析者）辅助 Agent 检查你的图片文件，记录其内容
- 对于缺失的资源：通过图像生成 API 自动生成（Gemini 或 xAI，取决于你的配置）
- 对 `SCENES.md` 中的每个场景：根据场景描述、美术方向和你提供的素材风格，生成场景目标参考图，写入 `references/scene_<name>.png`
- 用实际文件路径更新 `ASSETS.md`

**你得到什么：** `assets/` 目录里的美术文件、`references/` 目录里的场景参考图，以及一份已完全解析好路径、可供构建步骤使用的 `ASSETS.md`。这些场景参考图会成为视觉合约，后续 `/gm-evaluate` 用它们来对照运行时截图。

**需要知道的：** 如果你在 `ASSETS.md` 里添加了新资源或替换了图片文件，可以在任意里程碑重新运行 `/gm-asset`，它只会处理有变化的部分。

---

## `/gm-build`

**做什么：** 通过协调一组专门的辅助 Agent，实现整个游戏——包括所有 GDScript 代码、场景和单元测试。

**什么时候运行：** 在 `/gm-asset` 之后。当前里程碑必须已完成 `/gm-gdd`。

**背后发生了什么：**
- 恢复阶段：如果 `.godotmaker/verify_report.json` 中存在上一次 `/gm-verify` 留下的新鲜失败报告，把每个 check 的失败翻译成 `pending` 任务追加到 `PLAN.md`，再继续后续流程
- 读取 `PLAN.md`，找出待处理的任务，从风险最高的开始
- 派遣 Worker（执行者，最多同时 3 个）——每个 Worker 实现一个游戏系统及其单元测试，完成后汇报
- 大约每 5 个 Worker 之后，派遣一个 Verifier（验证者）——无界面编译项目并运行单元测试
- Verifier 通过后，派遣一个 Reviewer（评审者）——拥有 Godot 特有的领域知识（物理、UI、动画等），检查常见错误
- 如果 Reviewer 发现问题，新任务被追加到 `PLAN.md`，循环继续
- 只有当 `PLAN.md` 里所有任务都标记为 `verified`，且最后一轮评审没有发现新问题，构建才结束

**你得到什么：** `src/` 里的游戏代码、`scenes/` 里的场景、`tests/` 里的单元测试。

**需要知道的：** 在这个步骤里你无法自己写游戏代码——权限系统会阻止。主 Agent 负责协调，Worker 负责实际写代码。如果同一个任务失败三次，构建会暂停并询问你下一步怎么做。

---

## `/gm-verify`

**做什么：** 对整个项目做一次快速的机械检查——编译、单元测试、Lint、文件结构。

**什么时候运行：** 在 `/gm-build` 之后，以及每次 `/gm-fixgap` 之后。

**背后发生了什么：**
- 无界面运行 Godot 构建，检查编译错误
- 通过 `gdUnit4` 运行 `tests/` 里的所有单元测试
- 跑 `gdlint` 和 `gdformat --check` 做风格检查
- 通过 `tools/check_project.py --all` 跑静态项目检查
- 把结构化结论写入 `.godotmaker/verify_report.json`（每次都写，无论 PASS 还是 FAIL）

**你得到什么：** 两份产出——一份给人看的对话内验证报告，以及 `.godotmaker/verify_report.json` 同等信息的结构化版本。全部通过时，`/gm-verify` 还会向 `.godotmaker/stage.jsonl` 追加一个 `verify` 事件。

**需要知道的：** `verify_report.json` 是协议级别的反馈通道——下一步的 `/gm-build` 或 `/gm-fixgap` 读它,把失败翻译成待办任务,而不是盲目重试。每个 check 的 `result` 是 `pass | warn | fail | error` 四选一:`fail` = 项目代码有问题(修代码);`error` = 验证工具自身崩了,修法是改 lint/测试配置(`tooling_notes` 里给出),**不要删项目代码**。schema 和消费规则文档化在 `gm-verify/SKILL.md`。如果 `/gm-verify` 失败:构建阶段回 `/gm-build`,评估之后回 `/gm-fixgap`——两者都会自动读取这份报告。

---

## `/gm-evaluate`

**做什么：** 独立评估游戏是否达到了 GDD 承诺的标准，评估手段包括端到端测试和截图。

**什么时候运行：** 在 `/gm-verify` 通过之后。

**背后发生了什么：**
- 从零开始读取 `GDD.md`、`SCENES.md` 和 `PLAN.md`——对构建过程一无所知
- 在 `e2e/` 里写端到端测试，通过模拟玩家输入覆盖每个描述过的功能
- 运行这些测试，记录哪些通过、哪些失败
- 对每个场景截图，用视觉质量检查工具与参考图对比
- 得出最终结论：通过或拒绝，如果拒绝则附上具体问题列表

**你得到什么：** `.godotmaker/evaluation.json`（完整判定结果）以及 `e2e/screenshots/` 里的截图。

**需要知道的：** 评估者不能写游戏代码，也不能碰 `src/`——对游戏文件严格只读。被拒绝不是失败，而是信息。问题列表会直接被 `/gm-fixgap` 使用。

---

## `/gm-fixgap`

**做什么：** 读取评估报告的问题列表，派 Worker 逐一修复。

**什么时候运行：** 在 `/gm-evaluate` 拒绝构建之后。

**背后发生了什么：**
- 读取 `.godotmaker/evaluation.json` 拿到产品层问题；恢复阶段如果 `.godotmaker/verify_report.json` 有上一轮 verify 留下的失败，也一并读入
- 生成或合并 `GAP.md`——一个按优先级排列的任务清单：每个 `C`/`J`/`G` 字母分组内，verify 来源的机械层失败排在前面，evaluation 来源的产品层问题排在后面
- 派 Worker 处理每个关键和重要问题，采用和 `/gm-build` 相同的 Worker → Verifier → Reviewer 循环
- 将当前的 `GAP.md` 归档到 `.godotmaker/gaps/<n>/`，保留每次迭代的记录

**你得到什么：** 更新后的游戏代码，`GAP.md` 被移入归档。

**需要知道的：** `/gm-fixgap` 完成后，依次运行 `/gm-verify` 和 `/gm-evaluate`。这个循环持续下去，直到 `/gm-evaluate` 通过，或者你决定停下来。

---

## `/gm-accept`

**做什么：** 把通过评审的构建结果展示给你，记录你的决定。

**什么时候运行：** 在 `/gm-evaluate` 通过之后。

**背后发生了什么：**
- 向你展示最终评估摘要和截图
- 询问：接受、拒绝（返回 `/gm-fixgap`），还是停止
- 把你的答案记录在 `.godotmaker/stage.jsonl`

**你得到什么：** 一条已记录的接受事件，或者明确的指示返回重做。

**需要知道的：** 在这里拒绝是完全合理的——如果你在截图里看到不满意的地方，完全可以让它再跑一轮。你的决定永远是最终裁决。

---

## `/gm-finalize`

**做什么：** 干净地关闭当前里程碑——归档记录、写摘要、标记为完成。

**什么时候运行：** 在 `/gm-accept` 记录了接受决定之后。

**背后发生了什么：**
- 写入 `.godotmaker/final_report.json`，汇总构建了什么、完成了哪些任务、评估结果如何
- 归档里程碑文档（`GDD.md`、`PLAN.md` 等），避免被下一个里程碑覆盖

**你得到什么：** 一个已为下一个里程碑准备好的干净项目，当前里程碑的记录完整保留。

**需要知道的：** `/gm-finalize` 之后，如果想继续添加功能，下一步就是再来一个 `/gm-gdd`。`/gm-scaffold` 不需要再跑——项目已经存在了。

---

## 什么是里程碑？

一个里程碑 = 一轮完整的 `/gm-gdd` → `/gm-asset` → `/gm-build` → `/gm-verify` → `/gm-evaluate` → （fixgap 循环）→ `/gm-accept` → `/gm-finalize`。

第一个里程碑构建游戏核心。后续每个里程碑添加新的功能集或进行重大修改。这样你可以渐进地扩展游戏，不用从头开始，同时让每次 `/gm-build` 的体量保持在足够小、足够可靠的范围内。

要从全局视角了解各阶段是怎么衔接的，参见 [how-it-works.md](how-it-works.md)。
